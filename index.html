01 — Problem Statement

The Parallax framework enables cross-ECU signal communication via a Vehicle Model (RVM), but the full signal chain is buried across 4+ repositories and 3+ code generators. No single tool shows the complete path from a CAN frame to the Kotlin app that reads it.

02 — Solution Approach

AI-Powered Signal Traceability

Four capabilities that together make the invisible signal chain visible, queryable, and safe to refactor.

01 / 04

Multi-Layer Signal Parsing

Five purpose-built parsers extract structured data from every layer of the Parallax signal chain — no generated code or API needed.

  • RvmParser — extracts properties & services from *.rvm YAML, derives NATS subjects algorithmically
  • RoutingRulesParser — maps Elpis ↔ Parallax namespace from routing-rules/*.yaml
  • RalBusParser — captures PDU IDs, sender/receiver ECUs, cycle times from *.bus.ral
  • AppCodeParser — grep-based Kotlin analysis: buildFlow() / requestTyped() calls
  • OrchestratorParser — resolves fan-out: 1 service call → N downstream edges
L1 RVM Vehicle Model └─ namespace / property / service └─ owner ECU, scope └─ NATS subject (derived) L2 EGW2 Routing Rules └─ Elpis signal ↔ Parallax subject └─ direction: App→Zonal or reverse L3 RAL Bus Signals └─ PDU ID, CAN/ETH frame └─ sender ECU, receivers[] └─ cycle_time_ms L4 App Code (Kotlin) └─ buildFlow("STATE.*") └─ requestTyped("SERVICE.*") L5 Orchestrator └─ incoming_service └─ outgoing_services[6]
02 / 04

Graph-Based Flow Tracing

Parsed data populates a directed signal graph with nodes for signals, apps, ECUs and RVM entries — edges encode 3 confirmed flow patterns.

  • Pattern A — App → EGW2 → Zonal (command: e.g. setFanSpeed)
  • Pattern B — Zonal → EGW2 → App (state readout: e.g. fanSpeed)
  • Pattern C — App → Orchestrator → EGW2 × N (fan-out: setDriveMode → 6 services)
  • Graph stored in PostgreSQL as parallax_flow_edges with confidence scores
  • Cytoscape.js visualization with live traversal via MCP tool calls
App ──NATS──▶ RVM Node │ routing-rules YAML │ ▼ Elpis Signal (PDU) │ RAL: sender=XMMDROID receivers=[FZC, BZC] │ ▼ Zonal ECU hardware Fan-out (Pattern C): App ──▶ Orchestrator ├─▶ EGW2 ──▶ FZC ├─▶ EGW2 ──▶ BZC └─▶ EGW2 ──▶ CZC
03 / 04

AI Skills for Deep Analysis

KB AI skills wrap the signal graph with LLM-powered analysis — answering questions no SQL query can answer alone.

  • Signal trace skill — "Trace all signals in climate domain" → full dependency chain in seconds
  • Impact analysis — "What apps break if PDU 0x700 changes?" → backward graph traversal
  • Variant diff — Compare v1_0 vs v2_1 signal coverage: missing / new / changed entries
  • MCP tool — parallax_trace(signal) callable from IDE, CI, or chat interface
  • Git provenance — every signal linked to commit SHA + file + line for full auditability
// MCP Tool call parallax_trace("fan_speed") → { rvm_node: "climate.cabin.fan_speed", nats_subject: "STATE.VEHICLE.*.climate...", elpis_signal: "VWG_HVAC_UI_Status", pdu_id: "0x701", sender_ecu: "FZC", receiver_ecus: ["XMMDROID"], apps_subscribed:[ "HvacApp.kt:247", "ClimateWidget.kt:89" ], pattern: "B", confidence: 0.97 }
04 / 04

Impact & Value for the Team

Beyond a hackathon prototype — this fills a real gap in the development toolchain for Parallax-based vehicle software.

  • Hours → seconds: Signal archaeology that takes an engineer ~2.5 hrs is now a single MCP query
  • Safer refactoring: Full impact analysis before changing any routing rule or RVM property
  • Faster onboarding: New engineers explore the full signal flow interactively via graph UI
  • CI integration: Automated check "did this commit break a signal path?" via parallax_trace
  • Extensible: Plugin architecture adds new layers (Proto, SomeIP) without changing the core
Before (today): grep -r "VWG_HVAC_UI_Status" \ firmware/ elpis-gw-2/ \ routing-rules/ lagvm/ → ~2.5 hours, error-prone After (Parallax Tracer): parallax_trace("VWG_HVAC_UI_Status") → Full chain in < 2 seconds → Git SHA, line numbers, confidence scores, graph ROI per trace: ~2.5 hrs saved Signals in scope: 50+ (v1_0) Potential impact: 100s hrs/sprint
